# Git-msg Generator Skill

This skill helps teams standardize git commit messages with reusable rules, linting, and LLM-friendly templates. Scripts handle deterministic work (rule creation, file management, linting) while the LLM interprets rule intent, generates templates, and polishes messages.

## Quick start

- Create a rule interactively or with `--rule-name`:
  ```bash
  python scripts/init_rule.py
  python scripts/init_rule.py --rule-name RTOS
  ```
- Generate a context payload for the LLM:
  ```bash
  python scripts/generate_message.py --rule RTOS
  ```
- Lint a message for the same rule:
  ```bash
  python scripts/lint_message.py --rule RTOS --message "fix: update timer"
  ```

## Workflows

### 1. Rule creation
1. Run `scripts/init_rule.py` (optionally `--rule-name`).
2. Paste the natural-language description (with samples and requirements).
3. Name the rule, handle duplicates, and configure author/email defaults.
4. Opt into lint rule generation and decide if interactive output should be the default.
5. Wait for the LLM to populate `template.txt` and `lint-rules.json` inside `rules/<name>/` based on the stored spec and metadata.

### 2. Commit message generation
1. Stage your changes.
2. Run `scripts/generate_message.py --rule <name>` (pass `--author`/`--email` or let the rule supply them).
3. Feed the printed JSON payload to the LLM to get a line-by-line message that matches the stored template and requirements.
4. If interactive mode is enabled, include follow-up clarifications about DST/AR or semantic expectations.

### 3. Text linting
1. Produce or draft a commit message.
2. Run `scripts/lint_message.py --rule <name>` with `--message` or `--file`.
3. Fix any errors (format/semantic/length) reported by the script before committing.

## Rule layout

Rules live under `rules/<name>/`:

```
rules/<name>/
├── config.json          # rule metadata, author/email preferences, interactive flag
├── template.txt         # template generated by the LLM
├── lint-rules.json      # lint rules that the script enforces
└── .metadata.json       # internal state to track LLM work
```

`config.json` stores the raw description, default author/email, lint-enabled flag, and creation timestamp. `lint-rules.json` contains regex/length/format/semantic rules and is optional depending on the setup.

## Default assets

The skill ships with reference assets (see `assets/default-rules/`) and documentation to help teams bootstrap their own conventions. `RTOS` is the primary example with DST/AR and Signed-off-by requirements; `conventional` demonstrates a standard Conventional Commits setup.
